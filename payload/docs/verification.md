# Verification (Detailed)

Read when class is `normal-implementation` or above and you're designing acceptance probes, classifying verification evidence, or evaluating completion. The headline rules in `CLAUDE.md` (Verification Standard section) name this file as the source for the heavy machinery below.

## Acceptance-first verification
Derive checks from spec, not patch. Applicability: `tiny-fix` lightweight only. `normal-implementation`/`contract-surface-change`/`planned-multi-item`/`replacement-or-supersession`: universal parts (probe schema, paraphrase check, semantic class, executable proof-of-execution, counterexample question) per class; heavy parts (named artifacts, committed/tracked fixtures, two-reopening rule) at bars for `contract-surface-change`/in-scope `planned-multi-item`/`replacement-or-supersession`.
Define verification before or alongside implementation. For each item record: `source requirement` = exact phrase/behavior/invariant proven, quote user if available else cite issue/failing test/docs/inspected invariant/runtime observation; `positive probe` = behavior/route/artifact/state that must exist + proof (command/test/audit/runtime check/inspection); `negative probe` = old/direct/parallel/partial/forbidden path that must not exist + proof absent.
Positive probe must exercise **target-reachable path** users/target env uses: production endpoint for services, public API for libraries, CLI for tools, runtime loader for skills/plugins, install path for dotfiles, build/render path for docs; not helper/scaffold.
Full evidence must cover reachability ladder with executed probe or explicit `inspection` rationale: entry / validation / routing / execution / state effect / output / lifecycle handling / observability / regression test. Chain project-defined; assume no applicable link. Non-behavioral artifacts record "n/a — reason". Heavy/behavior-boundary lifecycle uses matrix below.
**Public-knob liveness example:** config field/CLI flag/mode/route label/env var traces `parsed → stored → affects eligibility/routing → affects execution → observable behavior → regression evidence` (executed regression or inspection-backed check naming what would change if broken; inspection okay for static config/local tools where executable regression disproportionate). Parses+stores but doesn't affect execution = scaffolding; `implemented_unverified` until all links evidenced.
**Negative probes required** when spec asserts binding semantic system claim: acceptance criterion, runtime invariant, replacement intent, universal coverage. Test: would removing word/phrase change what system must guarantee? Yes → binding claim/negative probe. No → procedural emphasis/style/no negative probe. Signal words (`all`/`every`/`only`/`no direct`/`same path`/`by construction`/`cannot`/`proves`/`supersede`/`replace`/`delete`/`remove`/`no longer`/contextually `must`/`always`/`never`) signal test, not automatic triggers.
**Behavioral matrix** for `contract-surface-change`/`replacement-or-supersession`/behavior-boundary `planned-multi-item` touching lifecycle/error handling/contract/persistence/concurrency/external integration: unsupported-input; ambiguous-input; missing-dependency; partial-availability; fallback; abort/cancel; cleanup (resource release after success/failure/partial); regression (existing behavior unaffected). Each matrix item is evidence-format probe. Cover touched boundaries only; not relevant → "n/a — reason." Lighter classes: recommended, not mandatory. Matrix supplies lifecycle evidence required by ladder for heavy classes.
Examples in request/issue/plan are minimum evidence, not full boundary. Expand requirement to implied semantic class and probe class, bounded by objective. Don't turn examples into new deliverables; if probe reveals outside-scope work, surface delta and request confirmation.
**Reusable abstraction probes:** if deliverable handles class wider than examples (generators, routers, classifiers, frameworks, templates, public reusable APIs, builder-style skills), run at least one fresh case not used in development. Scale with breadth/risk: internal helper one fresh case; broad public abstraction several cases across input dimensions including boundary/transfer. Canned/degenerate/non-fitting result fails; item remains `implemented_unverified` or `blocked` if no reasonable extension satisfies class. Representative examples necessary not sufficient.
Probes derive from recorded source requirement. Restating in own words risks narrowing; confirm probe catches violation of source requirement, not paraphrase.
**Same-pass probes provisional** for broad binding claims, replacement/deletion, safety-critical, universal-coverage. Before `verified` need `probe-recorded` (probe text recorded before completion; rewrites logged) and `counterexample-shown` (written check naming concrete failure probe catches: "if X broken, command fails with Y", citing command + expected error/diff, not "audit passed").
**Proof of execution mandatory** for every executable probe: actual command, exit code, truncated stdout/stderr OR captured-run path, timestamp, working dir. Prose doesn't satisfy. Can't execute (production/hardware unavailable) → `inspection` with rationale, residual gap with environment class, executed probe required at next reachable boundary. Invalid proof: synthetic traces, logs from another env, helper script mimicking entrypoint, release/readiness flags without executed probe.
Re-derivation alone insufficient. High-risk (broad/replacement/safety-critical/universal) audits must either (a) add tracked fixture/test when workflow accepts, OR (b) record negative fixture/documented forbidden example + exact failing command in evidence/ledger when tracked files noise. Audit reproducible from artifact, not prose. Lower-risk `normal-implementation` (narrow bug repairs larger than `tiny-fix`, not touching contract/replacement/safety boundaries): named artifacts/tracked fixtures recommended, not mandatory; probe schema, paraphrase check, semantic-class expansion, proof-of-execution, counterexample question still apply. `tiny-fix`: lightweight only.
Before completion claim per item answer: negative-probe hole (what old/parallel/forbidden path could still exist while checks pass?) and positive-probe hole (what required behavior absent/unproven while checks pass?). Material/in-scope/plausible/locally addressable hole → strengthen verification or keep open. Theoretical/out-of-scope/non-target/unrealistic/no-authority holes → residual gaps, don't block `verified`. Item `verified` only when probes pass; missing/incomplete/weaker probes → `implemented_unverified`.
**Diff review before `verified`:** separate from probes. VCS: diff against recorded base commit, not working tree alone. Outside VCS: recorded baseline (snapshot, hash, pre-change mtime, plus manual diff/inspection). Positive probe maps to diff lines introducing behavior, or responsible diff region/artifact delta + runtime evidence when single line meaningless. Negative probe maps to deleting lines OR absence-check (`rg` zero baseline and current) for preservation/non-regression. Replacement/deletion active removal: absence-check insufficient; need deletion evidence or before-present/after-absent. Incidental pass (unrelated diff, baseline content, concurrent merge) not evidence. No concrete change-evidence link → strengthen or reopen `implemented_unverified`.
**Environment:** run probes in target-equivalent env. If target is local/sandbox (local CLIs, repo dotfiles, dev tools, sandbox scripts, docs, user's-machine content), local is correct. If runtime/deploy differs materially (OS/arch/runtime/deps/network/cache/secrets), record `sandbox-only` and keep `implemented_unverified` until target pass. `sandbox-only` verifies only when target matches sandbox.
**Two-reopening rule:** item claimed complete then reopened because verification under-scoped original requirement (not user-expanded scope or normal debug before first claim) → after two reopenings on one item/deliverable, existing probes presumed compromised. Re-derive fresh: open source requirement first, write new probes before prior probes/diff/implementation, then consult prior artifacts for deltas. Prefer second agent; same-agent independence approximated by order.

## Verification cost model
Reuse one persistent build/verification target dir per worktree/branch; fresh dirs only if corrupted/invalid. Prefer batched verification at meaningful boundary; replacement batches at replacement boundary. Cap concurrent heavy builds competing for disk/memory/cache. Under environment pressure, use smallest checks proving boundary, defer redundant broader checks. Don't pay repeated multi-minute cold builds for low-leverage intermediary slices. Don't start heavy builds below safe disk/temp/memory headroom; repair first. Per worktree one live target dir + one live temp dir; replace only if corrupted/poisoned/proven invalid. Don't overlap cold builds for same graph unless information gain exceeds duplicated compile cost. Broad compile already proved relevant path builds → prefer smallest remaining exact test. Raw diagnostic volume is cost signal; if build still compiling/linking after boundary proven elsewhere, stop and switch. Language-specific rules in `language-rules.md`.
